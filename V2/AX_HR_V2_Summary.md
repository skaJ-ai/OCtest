## 1. 설계 원칙 요약

### TO-BE L6 표기 공식
```
[결과물 명사] + [L6 표준 동사]
```
- **결과물 명사**: 명단, 기준안, 기안서, 결과표, 안내문, 증빙서류 등 **실물** 명시 (데이터·내역·사항 등 추상어 배제)
- **L6 표준 동사 (v2.1 기준)** — L6 표기의 마지막 자리. 클렌징 분류(HR/DW/SSC) 도구. `완료`라는 단어 자체는 목록에서 제거됨 (L6 위치 자체가 마일스톤 완료를 내포하므로):

| 카테고리 | 동사 | 설명 |
|---|---|---|
| HR 판단형 | `확정` | 내부 버전 고정·최종 결정 |
| HR 판단형 | `승인` | 결재권자 공식 사인 |
| HR 판단형 | `발행` | 외부·타조직 공식 문서 생성 |
| HR·SSC 실행형 | `실시` | 사람이 직접 수행하는 1회성 실행 — HR(면담·행사·교육) + SSC(세팅·물리 작업). `개최` 통합. **앞 명사가 행위·이벤트 명사일 때만 사용** (문서·데이터 명사 뒤 사용 금지) |
| HR 실행형 | `작성` | 수기 문서·일지 완성 (시스템 없을 때) |
| DW 처리형 | `발송` | 메일·알림 개인 송신 |
| DW 처리형 | `배포` | 포털 게시·전체 공람 |
| DW 처리형 | `취합` | 복수 출처 데이터 집계 |
| DW 처리형 | `추출` | 시스템 조건 조회·뽑기 |
| DW 처리형 | `등록` | 시스템 신규 입력 |
| DW 처리형 | `접수` | 요청·신청 수령 |
| SSC 서비스형 | `제공` | 계약·기간 단위 서비스 운영 |

### 실행 주체 4분류
| 주체 | 기준 |
|---|---|
| **HR** | 대면 협상, 예외 의사결정, 최종 기준·명단 수립, 전략 기획 |
| **DW** | 규칙 기반 필터링, 템플릿 메일 발송, 시스템 등록·추출, 챗봇 응대 |
| **SSC후보** | 물리적 현장 세팅, 배송·발송, 시설 일상 운영 |
| **Eliminate** | 불필요 중간보고, 대시보드로 대체 가능한 수기 확인 |

### 계층 불변 규칙
- **L3**: 절대 불변 (10개 도메인 고정)
- **L4**: 원칙 불변 (예외적 통폐합만 허용)
- **L5**: 유연 변경 가능 (실제 흐름 반영)
- **L6**: 전면 재설계 (순수 결과물형 표기)

## 4. SME 검수 가이드 및 채점 루브릭

### 4-1. 이중 채점 구조 (핵심 원칙)

L6 주체 판단은 **동사 단독으로 불가능**하다. 결과물 성격이 1순위, 동사는 보조 힌트다.

| 우선순위 | 채점 항목 | 판단 질문 | 결과 |
|---|---|---|---|
| **1순위** | 결과물 성격 (판단 복잡도) | "이 결과물에 규칙 외 인간 판단이 필요한가?" | YES → HR / NO → DW 후보 |
| **2순위** | 동사 범주 | "이 L6 표준 동사는 어느 주체 카테고리인가?" | HR판단형→HR / DW처리형→DW / SSC→SSC |

### 4-2. 검수 3-질문 체계
검수자는 각 TO-BE L6 행에 대해 아래 3가지를 순서대로 판단한다.

| # | 질문 | Yes → | No → |
|---|---|---|---|
| Q1 | 이 마디 종료 시 **유형 결과물(명단/기안서/등록화면)** 이 남는가? | Q2 진행 | 표기 재수정 |
| Q2 | 판단·책임이 **사람(HR)**에게 있는가? | 주체 = HR | Q3 진행 |
| Q3 | 규칙 기반 자동화로 **DW**가 처리 가능한가? | 주체 = DW | SSC 또는 재분류 |

### 4-3. 재수정 빈도 높은 항목 유형

| 오류 유형 | 오류 표현 예시 | 올바른 표현 |
|---|---|---|
| 완료 suffix | "명단 취합 완료" | "명단 취합" |
| 동사성 명사 + 실시 오남용 | "안내 발송 실시" | "안내문 발송" |
| L5 명칭 반복 | L5: 타운홀 운영 → L6: "타운홀 운영 확정" | "타운홀 실시" |
| 추상어 결과물 | "기초 자료 취합", "현황 확인" | "수요조사 취합", "접속 현황 추출" |
| 단계 표현 | "1차 검증", "최종 확정" | 결과물 명사 + L6 표준 동사 |

### 4-4. 도메인별 우선 검수 순서
1. **제도** (159행, 복잡도 최고) — 다단계 승인 체계 재확인
2. **인력운영** (97행) — 부서간 조율 마디 주체 검토
3. **교육** (108행) — SSC후보 항목 그룹 B 최종 확정 포함
4. **총무** (46행) — SSC 전체 이관 범위 최종 승인
5. **채용·Sparse 6개** — 비교적 단순, 후순위 검수 가능

## 5. V1.0 대비 V2.0 변화사항

> 본 섹션은 V1.0 보고 이후 V2.0으로의 개선 근거를 설명한다.

### 5-1. V1.0의 구조적 문제

V1.0은 이전 LLM이 생성한 초안으로, 다음 4가지 구조적 결함을 가지고 있었다.

| # | 문제 | 사례 |
|---|---|---|
| **P1** | **기계적 2분할 템플릿** — 모든 AS-IS L6를 예외 없이 "DW 1차 검증 + HR 최종 확정" 2행으로 분할 | 강의장 세팅 → "운영 자원 취합(DW)" + "운영 시나리오 확정(HR)" |
| **P2** | **결과물 명사 추상어 남용** — 실물 명시 없이 "기초 자료", "현황", "내역" 등 추상어 사용 | "요건 확인용 기초 자료 취합", "최종 운영 시나리오 및 타임테이블 확정" |
| **P3** | **실행형 활동 표현 불가** — 면담, 행사, 교육 실시 등 HR 직접 실행 활동이 L6에 반영되지 않음 | 면담, 타운홀 등이 L5로만 존재, L6 분해 없음 |
| **P4** | **SSC 미정립** — 물리적 활동을 DW로 잘못 분류하거나, 횡단 묶음 이관 원칙 없이 산발 처리 | 강의장 세팅 → DW+HR 2행으로 분할 (SSC가 맞음) |

### 5-2. V2.0 핵심 개선사항

| 개선 항목 | V1.0 | V2.0 |
|---|---|---|
| 분해 방식 | 모든 항목 동일 2분할 | 5가지 패턴(A~E) 성격별 적용, 1~3행 유연 결정 |
| 결과물 명사 | 추상어 허용 | 실물 명사 엄격 적용 (추상어 금지 목록 관리) |
| 실행형 활동 | L6 표현 불가 | `실시`·`작성` 동사 신설로 면담·행사·수기 문서 표현 |
| SSC 처리 | 개별·산발 | 횡단 묶음 원칙 — 동일 성격 L6 전체를 한꺼번에 이관 |
| Sparse 도메인 | 템플릿 기계 적용 | L5 성격별 5가지 실행 유형(행사/면담/캠페인/운영관리/2트랙) 구분 |
| 동사 체계 | 10개 (미완) | 12개 체계화 — HR판단형(확정·승인·발행)·HR실행형(실시·작성)·DW처리형(발송·배포·취합·추출·등록·접수)·SSC서비스형(제공). `완료` 제거 |

### 5-3. Before / After 비교 사례

| AS-IS L6 | V1.0 (문제 사례) | V2.0 (개선) |
|---|---|---|
| 전체 채용규모 산정 | "요건 확인용 기초 자료 취합" (DW) + "타당성 검토 및 최종 결과안 확정" (HR) | "부서별 채용 수요조사 취합" (DW) + "전사 채용규모 확정" (HR) |
| 강의장 세팅 | "운영 자원 리스트 취합" (DW) + "최종 운영 시나리오 확정" (HR) | "강의장 세팅 실시" (SSC) — 1행으로 단순화 |
| 연봉계약서 메일링 | "수신 대상자 그룹 명단 추출" (DW) + "안내문 초안 톤앤매너 검수 및 확정" (HR) | "연봉계약서 안내 메일 발송" (DW) — 1행으로 단순화 |
| 타운홀 행사 (Sparse) | 부재 — L5로만 표현, L6 분해 없음 | "타운홀 계획서 확정" (HR) + "타운홀 안내 발송" (DW) + "타운홀 실시" (HR) + "결과 보고서 작성" (HR) — 행사형 4행 패턴 |
| 피트니스 운영 (총무) | "피트니스 운영 완료" (SSC) — "운영"이 지속 개념이라 L6에 부적합 | "피트니스 서비스 제공" (SSC) + "운영 결과 보고서 확정" (HR) — 서비스형+결과보고 분리 |

## 6. 설계 고민 과정 기록

> 주요 설계 의사결정의 배경과 해결 과정을 기록한다. SME 검수 시 "왜 이렇게 설계했는가" 질의에 대한 근거로 활용한다.

### 6-1. 실행형 활동을 L6로 표현하는 문제

**문제 인식**: 초기 공식 `[결과물 명사]+[완료 동사]`는 면담, 행사, 교육 실시처럼 **행위 자체가 가치**인 활동을 표현하지 못함. 강제로 "결과물"을 붙이면 왜곡 발생.

**검토 옵션**:
- 옵션 A: "면담 완료", "행사 완료" — 기존 `완료`를 모든 실행형에 적용
- 옵션 B: "실시" 동사 신설 — L6 표준 동사 목록에 추가

**결론 (옵션 B 채택)**:
- `완료`를 DW·HR·SSC 모두에 쓰면 주체 구분 기능이 사라짐 → 클렌징 도구로서 무력화
- 글로벌 BPM 표준(BPMN/APQC/ARIS)도 실행형 활동을 `[행위 명사]+[완료 내포 동사]`로 표현 → 국제 표준 부합 확인
- **`실시`**: HR·SSC 실행 활동 공용 (`개최` 통합)
- **`작성`**: 수기 문서 완성 전용 (시스템 없을 때)

### 6-2. "완료" 동사의 남용과 재정의

**문제 인식**: V1.0에서 완료가 SSC 외에도 산발 사용됨. 특히 총무 도메인 "피트니스 운영 완료"처럼 **지속적 운영 개념(L5 수준)**에 완료를 붙이는 것이 이질적.

**판단 기준 재정립**:
- `완료` = 공식 L6 표준 동사 목록에서 **완전 삭제**
  - 초기 중간 결론: SSC 1회성 물리만 허용 → 최종 결론: L6 위치 자체가 마일스톤 완료를 내포하므로 별도 `완료` 불필요
- SSC **1회성** 물리 작업 → `실시` (세팅·포장·배달 등)
- SSC **지속** 운영 서비스 → `제공` 신설 (계약·기간 단위)
- "피트니스 운영 완료" → "피트니스 서비스 제공" (SSC) + "운영 결과 보고서 확정" (HR) 분리

### 6-3. SSC 시설 운영의 L5 vs L6 경계 문제

**문제 인식**: 총무 도메인에서 "운영"은 수개월~연단위 지속 활동으로 L5 수준 개념. 이것을 L6 단일 마디로 표현하면 시작·종료 경계가 없어짐.

**해결**:
- L5 = 총체적 운영 영역 (예: "피트니스 센터 운영")
- L6 = 경계 있는 단위 (계약/기간 기준): "피트니스 서비스 제공" (SSC)
- HR은 운영 결과에 대한 판단만: "운영 결과 보고서 확정" (HR)

### 6-4. ER·노사 영역 주체 원칙 확립

**문제**: Sparse 도메인 추론 시 DW 자동화 가능 여부를 먼저 판단하면, ER·노사의 면담·협의 활동까지 DW로 분류될 위험.

**원칙 확정**: ER·노사 영역은 **가급적 최대한 HR 주체**로 반영.
- 휴먼터치(공감·대면 협의)는 DW로 대체 불가한 고부가가치 활동
- DW 억지 분할 금지 (취합만 DW 허용, 판단·실행은 HR)

### 6-5. 최종 L6 표준 동사 체계 확정 (V2.1)

기존 10개 동사에서 13개로 정비. 추가/변경 내역:

| 변경 유형 | 동사 | 내용 |
|---|---|---|
| **신규 추가** | `실시` | HR·SSC 실행형 — `개최` 통합. "사람이 직접 수행하는 1회성 실행"으로 주체 확장 |
| **신규 추가** | `작성` | 수기 문서·일지 전용 (`등록`의 수기 버전) |
| **신규 추가** | `제공` | SSC 서비스형 — 지속 운영 활동 표현 |
| **제거** | `완료` | 공식 L6 표준 동사에서 삭제 — 모든 동사는 완료를 이미 내포하므로 불필요. SSC 물리형 → `실시`로 대체 |
| **제거** | `개최` | `실시`로 통합, 독립 동사 사용 금지 |

> 나머지 9개 동사(확정/승인/발행/배포/발송/취합/추출/등록/접수)는 현행 유지. 유사 쌍 검토 결과, 배포(포털 게시)와 발송(메일 push)은 메커니즘이 달라 통합 불가 판단.
> **최종 공식 동사: 12개** (확정·승인·발행·실시·작성·발송·배포·취합·추출·등록·접수·제공)
