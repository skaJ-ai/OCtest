# V2 실행 계획 (PLAN)

> CONTEXT → INVENTORY → **PLAN** 순서로 읽을 것.
> 실제 CSV 작성 전 이 문서 전체를 숙지하고 시작한다.

---

## 1. 출력 파일 스펙

### 1-1. 파일 목록

```
V2/
├── CONTEXT.md              ✅ 완료
├── INVENTORY.md            ✅ 완료
├── PLAN.md                 ✅ 완료
├── AX_HR_V2_Master.csv     ← 메인 산출물
└── AX_HR_V2_Summary.md     ← CSV 완성 후 작성
```

### 1-2. CSV 컬럼 정의

```
[AS-IS] L3, [AS-IS] L4, [AS-IS] L5, [AS-IS] L6,
[TO-BE] L3, [TO-BE] L4, [TO-BE] L5, [TO-BE] L6,
주체, 전략 사유, 분해유형
```

| 컬럼 | 설명 | 규칙 |
|---|---|---|
| [AS-IS] L3 | 원천 도메인 | realdata.ts 그대로 |
| [AS-IS] L4 | 원천 프로세스 그룹 | realdata.ts 그대로 |
| [AS-IS] L5 | 원천 단위 프로세스 | realdata.ts 그대로 |
| [AS-IS] L6 | 원천 활동 | realdata.ts 그대로. Sparse는 `(데이터 없음)` |
| [TO-BE] L3 | 목표 도메인 | AS-IS L3와 항상 동일 |
| [TO-BE] L4 | 목표 프로세스 그룹 | 원칙 AS-IS 유지. 예외 시 사유 명시 |
| [TO-BE] L5 | 목표 단위 프로세스 | 업무 성격에 따라 변경·분리 가능 |
| [TO-BE] L6 | 목표 성과 활동 | `[결과물 명사] + [상태동사]` 공식 적용 |
| 주체 | 실행 주체 | `HR` / `DW` / `SSC` / `Eliminate` |
| 전략 사유 | 주체·분해 판단 근거 | 1~2문장. 통폐합 시 "통폐합: [A]+[B]" 명시 |
| 분해유형 | 매핑 방식 | 아래 코드 참조 |

### 1-3. 분해유형 코드

| 코드 | 의미 |
|---|---|
| `1:1` | AS-IS L6 1개 → TO-BE L6 1행 (명칭만 결과물형 변환) |
| `1:2` | AS-IS L6 1개 → TO-BE L6 2행 분해 |
| `1:3` | AS-IS L6 1개 → TO-BE L6 3행 분해 |
| `통폐합` | 여러 AS-IS L6 → TO-BE L6 1행으로 통합 |
| `L5→L6` | Sparse: AS-IS L5가 TO-BE L6로 내려옴 |
| `SSC묶음` | 횡단 SSC 이관 대상 (전 도메인 작성 후 최종 확정) |
| `Eliminate` | 해당 L6 제거 |

## 2. 작업 원칙

### 2-1. 행 작성 순서
1. AS-IS L3/L4/L5/L6 그대로 입력
2. TO-BE L3 = AS-IS L3 (복사)
3. TO-BE L4 판단 (원칙 유지, 예외 시 사유 기록)
4. TO-BE L5 판단 (업무 성격 기반, CONTEXT 패턴 참조)
5. 패턴 선택 (A/B/C/D/E)
6. TO-BE L6 작성 (`[결과물 명사] + [상태동사]`)
7. 주체 배정
8. 전략 사유 작성
9. 분해유형 코드 입력

### 2-2. 1개 AS-IS L6가 여러 TO-BE 행이 될 때
- 동일 AS-IS L6를 여러 행에 반복 기입
- 분해유형에 `1:2` 또는 `1:3` 표시

### 2-3. 여러 AS-IS L6가 1개 TO-BE 행으로 합쳐질 때 (통폐합)
- 대표 AS-IS L6를 첫 행에 기입
- 나머지 AS-IS L6는 별도 행에서 TO-BE L6를 통폐합된 행과 동일하게 기입
- 분해유형 `통폐합`, 전략 사유에 통합된 항목 명시

### 2-4. SSC 횡단 묶음 처리
- CSV 작성 중에는 주체를 `SSC(후보)`로 표기, 분해유형 `SSC묶음`
- 전 도메인 완성 후 INVENTORY 섹션 5 대조 → 묶음 확정 시 `SSC`로 확정
- 단독 L6만 존재하고 묶을 유사 항목이 없으면 `HR` 또는 `DW`로 변경

## 3. 도메인별 핵심 판단 포인트

### 채용

| L5 | 핵심 판단 |
|---|---|
| 채용 계획 수립(경영계획) | 전체/팀별/사업부 L6 4개 → Pattern A. 사업부 채용규모 확정은 통폐합 검토 |
| 직무기술서 작성 | L6 1개 → 1:1. 요청 발송(DW) |
| 인재 소싱 채널 선정 및 액팅 | 발굴 기준 확정(HR) + 채널 선정(HR) + 후보풀 등록(DW). TO-BE L5명 간소화 검토 |
| 서류심사 | V1.0 "지원서 필터링 및 평가" L5 분리 채택 검토. 결과안내는 별도 TO-BE L5 |
| 자격 검증 | 대상자 확정(HR) + 일정 안내(DW) + 장소준비(SSC후보) + 결과 등록(DW) |
| 면접 | 위원 확정(HR) + 일정 안내(DW) + 장소준비(SSC후보) + 평가 취합(DW) |
| 채용 및 처우 승인 | 처우 협의는 HR 1행(대면). 품의 등록은 DW. 오퍼레터 발송은 DW |
| 선확보 인력 관리 | 등록(DW) + 평가 취합(DW) + 리텐션 방안 확정(HR) |
| 입사 및 온보딩 | 입사절차 안내(DW) + 발령 등록(DW) + 웰컴키트 발송(SSC후보) + 멘토 배정 확정(HR) |

### 인력운영

| L5 | 핵심 판단 |
|---|---|
| 사내 Job Posting | L6 13개로 많음. L5 분리 검토: "Job Posting 전형" / "전배 발령 처리" |
| 개별 전배 | Pattern A. 충원 타당성(HR) → 인력 서칭(DW) → 전배 발령 등록(DW) |
| 휴직/복직 | SSC 후보 검토 중. 면담은 HR 1행. 발령 등록은 DW. 안내 발송은 DW |
| 퇴직 | 면담은 HR 1행. 핵심인력 변제 확인(DW) + 요청 발송(DW) |
| 리텐션 | Pattern C. 퇴직 사유 데이터 추출(DW) → 분석·방안 확정(HR) |
| 인력 계획 | L6 11개. L5 분리: "인력계획 수립" / "인력계획 공지 및 등록". 수정경영계획은 Sparse 처리 |
| 제증명 관리 | 신청 접수(DW, 챗봇) → 예외 검토(HR) → 발급 등록(DW) |
| 인사발령 | 대상자 추출(DW) → 보고서 확정(HR) → 발령 공지 발송(DW) |

### 제도

| 공통 L5 패턴 | 표준 분해 |
|---|---|
| 가이드 수립/배포 및 시스템 준비 | 현황 추출(DW) → 가이드 확정(HR) → 가이드 배포(DW) + 시스템 셋팅 등록(DW) |
| 대상자 확정/변경 | 예외 사유 추출(DW) → 대상자 명단 확정(HR) → 명단 등록(DW) |
| 승인자/부서장 매핑 확정/변경 | 매핑 현황 추출(DW) → 매핑 명단 확정(HR) |
| 운영/임직원 소통 | 안내문 확정(HR, 민감 시) → 안내문 발송(DW). 독려는 DW 1행 |
| 결과 분석/보고 | 실시율 추출(DW) → 결과 보고서 확정(HR) |
| 후속 조치 | 항목별 성격에 따라 A 또는 B |

| L4 | 핵심 판단 |
|---|---|
| 업적평가 | 본인 평가 입력은 임직원이 주체 → DW(시스템) 처리. 평가면담은 HR 1행 |
| 승격 | 사령장 배포는 Pattern B (DW). 승격발령 반영은 DW |

| 계약직/변호사 관리 | 계약서 관리(DW) + 처우 결정(HR) + 재계약 통보(DW) |

### 교육

| L5 | 핵심 판단 |
|---|---|
| 교육 실행 계획 수립 | L6 1개 → 1:2. 현황 추출(DW) + 연간 계획 확정(HR) |
| 대상자 모집/선발 | 모집 공지(DW) + 선발 기준 확정(HR) + 합격 안내(DW) |
| 운영준비(인프라 셋팅 등) | L6 7개. 강의장 세팅·PC 대여·다과·교보재·발송 → SSC후보(횡단묶음). 체크리스트·시설 예약은 DW |
| 과정 운영 | 강사 영접·출석 체크(DW or SSC후보) + 강의 진행은 HR/강사 → 주체 표기 주의 |
| 과정설문 취합/분석 | 설문 시스템 개설(DW) + 설문 결과 추출(DW) + 분석 보고서 확정(HR) |
| 성과분석/결과보고 | 결과 보고서 확정(HR) + 강사 감사레터 발송(DW) |
| 내부 컨텐츠 개발 | L6 6개. 모듈계획서 확정(HR) + 강의안 확정(HR) + 교보재 제작 발주(DW) |
| 컨텐츠 개발(외부) | 강사 탐색·섭외(HR) + 계약 체결(HR) + 이러닝 개발 계약(DW) + 검수(HR) |

### Sparse 도메인 공통 접근

```
1. L5명을 보고 업무 성격 판단 (의사결정 / 데이터처리 / 물리적 운영)
2. 해당 성격에 맞는 Pattern (A/B/C/D) 선택
3. L6 2~3개 추론:
   - DW형: [L5 관련 현황] 추출 or 취합
   - HR형: [L5 핵심 결과물] 확정 or 승인
   - 발송형: [결과 안내문] 발송
4. 도메인 성향 반영 (INVENTORY 3절 참조)
```

**Sparse 특이 케이스**

| 도메인 | 특이 L5 | 판단 |
|---|---|---|
| 보상/근태 | 법정복리후생 (국민연금, 고용보험 등) | 신고 기한 기반 DW 자동화. 예외만 HR |
| 노사 | 단체교섭, 쟁의행위 | HR 전담 (대면 협상) |
| 노사 | My Pulse 서베이 | 서베이 발송(DW) + 결과 분석(HR) |
| 총무 | 파견자 문의 대응 | DW 챗봇 1행 |
| 해외인사 | M&A 실사 | Pattern C. 문서 추출(DW) + 분석 보고(HR) |
| 기획 | People팀 주간업무 작성 | Eliminate 검토 (관행적 수기 보고) |
| 임원조직 | 임원 피드백 면담 | HR 1행 (면담은 분할 불가) |

## 4. 품질 체크리스트

CSV 작성 중 및 완성 후 아래를 점검한다.

### 4-1. 행 단위 체크 (작성 중)
- [ ] TO-BE L6가 `[결과물 명사] + [상태동사]` 형식인가?
- [ ] 결과물 명사가 추상어(데이터, 내역, 현황 등)가 아닌 실물인가?
- [ ] CONTEXT 금지 패턴이 TO-BE L6에 없는가?
- [ ] `완료` 동사를 DW·HR 행에 쓰지 않았는가?
- [ ] `취합`과 `추출`을 상황에 맞게 구분했는가?
- [ ] 면담·협의 활동을 DW로 분할하지 않았는가?

### 4-2. L5 단위 체크 (L5 완료 후)
- [ ] 분해 행 수가 업무 성격에 맞는가? (과도한 팽창 없는가?)
- [ ] 통폐합 항목의 전략 사유에 원래 항목이 명시되어 있는가?
- [ ] TO-BE L5 분리 시 L4보다 큰 단위가 되지 않는가?

### 4-3. 도메인 완료 후
- [ ] 동일 도메인 내 유사 L6 표현이 일관성 있게 작성되었는가?
- [ ] SSC(후보) 표기 항목이 누락 없이 INVENTORY 5절과 일치하는가?
- [ ] Eliminate 항목에 명확한 근거가 전략 사유에 있는가?

### 4-4. 전체 완성 후
- [ ] SSC(후보) 전체 검토 → 횡단 묶음 확정 → `SSC`로 최종 확정
- [ ] 도메인별 행 수가 INVENTORY 예상 범위(±30%)에 있는가?
- [ ] TO-BE L5 변경 사항 목록 정리 → Summary.md에 반영
- [ ] 주체별 비율 집계 → Summary.md에 반영

```

## 4. SSC 횡단 묶음 확정 목록

```

## 6. 작업 시작 전 최종 확인

1. **CONTEXT.md** 금지 패턴 9개 암기 수준으로 숙지
2. **INVENTORY.md** 도메인별 L5 목록 파악
3. **이 PLAN.md** 도메인별 판단 포인트 확인
4. CSV 헤더 한 번에 작성 후 채용 도메인부터 순서대로 진행
5. 도메인 하나 완성할 때마다 INVENTORY 작업 현황 ✅ 업데이트
